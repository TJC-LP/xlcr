# Introduction
Transitioning from a **Kotlin-based MCP implementation** to a **Java-based MCP stack** for Excel and PowerPoint requires careful planning of architecture, interface design, integration, and performance validation. The goal is to reimplement the _Model Context Protocol (MCP)_ server in Java to support real-time editing of Excel spreadsheets and PowerPoint presentations, while interoperating with the existing Scala-based XLCR core (the collaboration/calculation pipeline). This report provides practical implementation guidance focusing on four key deliverables: (1) architecture diagrams of the new Java MCP stack, (2) Java interface definitions for Excel and PowerPoint document models, (3) integration patterns between the Java server and the Scala XLCR core, and (4) a performance benchmarking strategy. Each section is organized with diagrams or code examples and clear recommendations to support engineering execution.

## 1. Architecture and Component Interaction
**Component Overview:** The Java MCP server will be structured in layered components handling transport, protocol, document logic, and external integration. At a high level, clients (which could be user interfaces or AI agents) connect via supported **transport interfaces** – WebSockets for full-duplex communication, Server-Sent Events (SSE) for one-way event streams, or standard I/O (stdio) for local subprocess control. These feed into the MCP **protocol layer** and **session manager**, which decode protocol messages, manage session state, and dispatch user actions to appropriate handlers. For each document type (Excel or PowerPoint), a **document handler** manages an in-memory **document model** (backed by Apache POI for Office file parsing/manipulation). **Action handlers** encapsulate logic for various editing actions (e.g. editing a cell, inserting a slide) and apply these to the document model. The server integrates with the **XLCR Scala core** for any heavy computations or cross-session coordination (for example, formula calculation or business logic pipelines), either via direct in-process calls or an IPC/RPC mechanism. The diagram below illustrates these components and their interactions in the Java MCP architecture:

([image]()) *Architecture of the Java-based MCP server for Excel and PowerPoint.* The client connects via WebSocket, SSE, or stdio transports into the MCP Protocol/Session layer, which dispatches incoming actions to document-specific handlers. Excel and PowerPoint document handlers use Apache POI-backed models to apply changes and track mutations. Action handlers encapsulate edit operations. The server can invoke the Scala-based XLCR core for advanced processing or synchronization as needed. Document events (updates) flow back through the protocol layer to notify all clients in real time.

**MCP Protocol & Transports:** The MCP protocol layer is responsible for real-time messaging between clients and server. It maintains sessions (e.g. one session per open document collaboration) and ensures each incoming message is routed to the correct document handler. Communication can use WebSockets or SSE depending on client capabilities. WebSockets provide bidirectional, low-latency channels suitable for interactive editing (the server can receive edits and push updates instantly) ([SSE vs WebSockets: Comparing Real-Time Communication Protocols](https://softwaremill.com/sse-vs-websockets-comparing-real-time-communication-protocols/#:~:text=What%20is%20WebSockets)) ([WebSockets vs Server-Sent Events: Key differences and which to use in 2024](https://ably.com/blog/websockets-vs-sse#:~:text=,been%20supported%20by%20all%20major)). SSE provides a simpler one-way channel from server to client – useful for broadcasting updates to clients that only need to receive changes (e.g. a live viewer) ([WebSockets vs Server-Sent Events: Key differences and which to use in 2024](https://ably.com/blog/websockets-vs-sse#:~:text=,been%20supported%20by%20all%20major)). Both protocols use a single long-lived connection for continuous updates, reducing overhead of HTTP polling. The Java MCP server can offer both: for full editing clients, WebSocket is used to send edits and receive updates; for read-only or event-driven clients, SSE can stream the document events. A stdio transport interface is also included to allow local processes (such as AI agents running as subprocesses) to send/receive MCP messages via standard input/output, making integration with other tools straightforward. The protocol layer likely uses a JSON or similar message format to represent actions and events. It also handles session initiation (e.g. client requests to open a document), heartbeats or pings, and session termination.

**Document Handlers and Session Lifecycle:** When a session is created (e.g. a user opens an Excel file), the protocol layer instantiates the appropriate **document handler** – `ExcelDocumentHandler` or `PowerPointDocumentHandler` – in Java. This handler loads the document from storage (using Apache POI to parse the `.xlsx` or `.pptx` file into memory) and creates a **document model** object representing the workbook or presentation. The session manager keeps track of the association between session ID, clients, and the open document handler. Real-time editing follows a typical lifecycle: a client’s edit action is received via the protocol layer, which identifies the target session/document and invokes the corresponding action handler. For example, an “edit cell” action on an Excel session will be dispatched to an Excel action handler method. The action handler, running on the server, calls into the Excel document model’s API to apply the change (e.g. update the cell value, modify formatting, etc.). The document handler can then broadcast the change to all other connected clients in that session via the protocol (ensuring everyone’s view stays in sync). This cycle repeats for each edit. The session manager ensures thread-safe sequencing of operations per document (e.g. by queueing actions or using locks on the document model) to avoid race conditions if multiple clients edit concurrently. It also handles lifecycle events: when the first client opens a document, load it; when the last client leaves, optionally save and close the document, freeing resources. Real-time collaboration is thus achieved by the server acting as a hub: each edit is immediately applied to the server’s authoritative document model and then propagated to other participants. The use of persistent sessions also allows maintaining context such as undo history or user-specific data on the server side.

**Real-Time Editing Flow:** To visualize the flow, consider a typical sequence for an Excel editing session:

1. **Session Initiation:** A client requests to open an Excel file (via a protocol message). The MCP server session manager creates a new session and uses the Excel document handler to load the workbook (e.g. via `XSSFWorkbook` for .xlsx) into an in-memory model. It might also invoke the Scala XLCR core if any initial processing (like pre-calculating formulas or loading shared state) is needed. The server then sends an “open successful” event back to the client, possibly including an initial JSON snapshot of the document structure so the client can render it.

2. **User Edit Action:** The user edits the document (e.g. changes a cell value). The client sends an MCP message (over WebSocket or stdio) describing the action (e.g. “update cell A1 to 42”). The MCP protocol layer receives this JSON message from the socket stream and routes it to the Excel action handler in the server. That handler interprets the action, then calls the Excel document model’s method (e.g. `setCell(sheetName, row, col, value)`). The document model (backed by POI) updates the in-memory workbook and records the mutation (for change tracking). If the action triggers any computations (like formula recalculation or slide layout adjustment), the handler may call out to the XLCR core or use POI’s built-in capabilities (for instance, POI’s `FormulaEvaluator` can calculate Excel formulas on the fly ([Formula Evaluation - Apache POI](https://poi.apache.org/components/spreadsheet/eval.html#:~:text=The%20POI%20formula%20evaluation%20code,in%2C%20or%20created%20in%20POI))). On success, the document handler notifies the protocol layer of the updated state – for example, which cell changed and its new value.

3. **Broadcast Update:** The MCP protocol layer formats an update event (e.g. JSON diff or new cell value) and sends it out to all subscribed clients in the session. For WebSocket clients, it pushes a message; for any SSE clients, it sends an event in the text/event-stream. The originating client might receive an acknowledgment as well. Now all collaborators see the change in real-time. This event-driven broadcast is handled quickly – the persistent WebSocket/SSE connections allow server->client push with minimal latency (no HTTP re-connect overhead) ([SSE vs WebSockets: Comparing Real-Time Communication Protocols](https://softwaremill.com/sse-vs-websockets-comparing-real-time-communication-protocols/#:~:text=What%20is%20WebSockets)) ([WebSockets vs Server-Sent Events: Key differences and which to use in 2024](https://ably.com/blog/websockets-vs-sse#:~:text=,been%20supported%20by%20all%20major)). The system targets low latency (e.g. tens of milliseconds) from the moment a user makes an edit to the moment others see it.

4. **Continuous Editing & Synchronization:** The process repeats for subsequent edits. The session manager might enforce an ordering if needed (e.g. queue actions per session to ensure the document model processes one at a time). If multiple users or an AI and a user simultaneously issue conflicting edits, the server can serialize them or apply last-write-wins; more advanced conflict resolution (like operational transform or CRDT) could be integrated in the future if true simultaneous multi-edit on the exact same content is needed. (In many cases, Excel or PPT collaboration might lock at a worksheet or slide level to avoid conflicts.) The server ensures each action is fully applied and broadcast before the next, maintaining consistency.

5. **Session Termination:** When editing is done (users close the document or after an idle timeout), the session manager triggers cleanup. The Excel/PowerPoint handler can save the final document state back to persistent storage (using POI to write out the modified file). All resources for that session (in-memory workbook, slide deck, etc.) are released to prevent memory leaks. The server may log the session’s activity or send a final summary.

Throughout this flow, **lifecycle management** is crucial. The architecture should handle clients joining or leaving mid-session (new clients might need an initial full document snapshot), recovery from errors (e.g. if an action fails, send an error message and possibly rollback if supported), and efficient resource usage (e.g. do not keep large POI objects in memory longer than needed). The diagram and description above highlight how the Java MCP server orchestrates real-time editing by combining transport protocols, a robust session manager, document-specific handlers, and integration with external services for extended functionality.

## 2. Document Model Interfaces (Excel & PowerPoint)
A core part of the implementation is designing **Java interfaces for the document models** that represent Excel workbooks and PowerPoint presentations. These interfaces abstract the underlying Apache POI library and provide methods for mutation, introspection, and data transformation. By defining clear interfaces, we ensure a clean separation between the MCP server logic and the document manipulation details, allowing easier maintenance and potential extensibility (for example, adding a Word document model later or swapping out the POI library if needed).

**Design Goals:** The document model interfaces should support: (a) **mutation tracking** – the ability to record what changes have been made for broadcasting or later auditing; (b) **JSON transformation** – converting document content or changes into a JSON structure that can be sent over MCP to clients or used by other components; (c) **preview generation** – producing a lightweight preview of the document (e.g. an image thumbnail of a slide, or text summary of a spreadsheet) for quick display; and (d) **structure introspection** – querying the structural elements of the document (sheets, slides, shapes, charts, etc.). Additionally, they should leverage Apache POI’s capabilities for reading/writing Office formats and be designed for extensibility (e.g. using interfaces/abstract classes that can be implemented differently if needed).

Below are example interface definitions in Java that meet these requirements. We define a generic `DocumentModel` base interface for common operations, and specialized interfaces `ExcelDocumentModel` and `PowerPointDocumentModel` for type-specific functionality. (In practice these could be classes with a common abstract parent, but here we show them as interfaces for clarity of the contract.)

```java
public interface DocumentModel {
    /** Serialize the entire document (or a delta of changes) to a JSON structure. */
    String toJson(); 

    /** Generate a preview of the document (e.g., image or text snippet). */
    byte[] getPreview(); 

    /** Get a representation of the document's structural elements (for introspection). */
    DocumentStructure getStructure(); 

    /** Get and clear the list of recent changes (mutation log) since last checkpoint. */
    List<Change> pullRecentChanges(); 

    /** Save the document model back to a file (persist changes). */
    void save(Path filePath) throws IOException; 

    /** Close and release resources associated with this document. */
    void close(); 
}

// Example structure classes for introspection and changes:
public class DocumentStructure { /* e.g., for Excel: list of sheets; for PPT: list of slides */ }
public class Change { 
    String type; // e.g., "CellUpdate", "SlideAdded"
    Object detail; // e.g., cell reference and new value, or slide index
    /* timestamp, user, etc. could be included */ 
}
```

```java
public interface ExcelDocumentModel extends DocumentModel {
    /** Get the names of all worksheets in the workbook. */
    List<String> getSheets(); 

    /** Retrieve a cell's value (and possibly formula) given sheet name and coordinates. */
    CellData getCell(String sheetName, int rowIndex, int colIndex);

    /** Set or update a cell’s value or formula, returning a Change record. */
    Change setCell(String sheetName, int rowIndex, int colIndex, Object value);

    /** Insert a new row or column (mutation tracked internally). */
    Change insertRow(String sheetName, int rowIndex);

    /** Delete a row or column. */
    Change deleteRow(String sheetName, int rowIndex);

    /** Apply a style or formatting change to a range of cells. */
    Change formatCells(String sheetName, CellRange range, CellStyle style);

    // ... additional Excel-specific operations as needed (rename sheet, add chart, etc.)
}
```

```java
public interface PowerPointDocumentModel extends DocumentModel {
    /** Get a list of slides with their titles or IDs. */
    List<SlideInfo> getSlides(); 

    /** Add a new slide (with given layout/template) at a specified position. */
    Change addSlide(int index, SlideLayout layout);

    /** Remove a slide by index. */
    Change removeSlide(int index);

    /** Edit text content in a text box shape on a slide. */
    Change updateText(int slideIndex, String shapeId, String newText);

    /** Add an image to a slide. */
    Change addImage(int slideIndex, ImageData image, Position position);

    // ... additional PowerPoint-specific operations (change layout, transition effects, etc.)
}
```

In the above interfaces, each method that mutates the document returns a `Change` object, which is a simple record of what was done. The `DocumentModel.pullRecentChanges()` could accumulate these changes and allow the server to retrieve and broadcast them in batch or individually. This is one way to implement **mutation tracking** – by logging changes as `Change` objects whenever a setter method or an insert/delete is called. Another approach could be maintaining a version counter and a diff, but using explicit change objects is straightforward for capturing discrete user actions.

**JSON Transformation:** The `toJson()` method in `DocumentModel` would produce a JSON string (or potentially a structured object that can be serialized to JSON) representing either the entire document or recent changes. For Excel, this might output a JSON with workbook metadata and sheets as arrays of cell values (possibly truncated for performance). For example, an Excel workbook’s JSON could look like: `{ "sheets": [ { "name": "Sheet1", "cells": [ [null, "Hello"], [5, "=SUM(A2:A5)"], ... ] }, ... ] }`. For PowerPoint, `toJson()` might list slides and their elements (text, images, etc.) in a hierarchical JSON. This allows the client or other systems to consume document data easily. Apache POI can be used to iterate over all cells or slides to build this JSON. The design should consider that large documents might produce huge JSON strings – so in practice we might use pagination or diff-based JSON (only changes) after initial load. Still, the interface provides the capability, and we can optimize under the hood (e.g. `toJson()` could accept an option for full vs. diff export).

**Preview Generation:** The `getPreview()` method returns a `byte[]` (which could be an image file bytes or other format) for a quick preview. For an Excel spreadsheet, a “preview” might be an image of the first sheet or a small HTML snippet showing a portion of the data. Apache POI doesn’t directly render Excel to image, but one could generate HTML or use a third-party rendering. A simpler approach is to create a CSV or text excerpt of the top of the sheet as a preview. For PowerPoint, Apache POI **does** support rendering slides to images – it provides a utility (PPTX2PNG) that can convert slides to PNG pictures ([Rendering slideshows, WMF, EMF and EMF+](https://poi.apache.org/components/slideshow/ppt-wmf-emf-renderer.html#:~:text=For%20rendering%20slideshow%20,an%20utility%20class%20%20PPTX2PNG)). We can leverage that in the implementation: e.g. use POI’s `SlideShow` API to draw a slide onto a `Graphics2D` and produce an image. In our interface, `getPreview()` for PowerPoint might return a PNG of the first slide (or a thumbnail strip of all slides). This helps when, for instance, an AI agent or a user interface wants a quick look at the content without transferring the entire file. The interface keeps it general (just returns bytes – the calling code could interpret it as needed, perhaps along with a MIME type or an `Image` object instead).

**Structure Introspection:** The `getStructure()` method returns a `DocumentStructure` object that outlines the high-level structure of the document. For Excel, this could include the list of sheet names, each sheet’s dimensions (number of rows/cols), maybe presence of charts or pivot tables. For PowerPoint, it could list slide titles, layout types, and any embedded media. This is useful for both clients (to build an outline or navigator) and for the server or other services (to understand the document without reading everything). Apache POI can provide this info: e.g. `Workbook.getNumberOfSheets()` and `Sheet.getSheetName()` for Excel, or `XMLSlideShow.getSlides()` for PPT to list slides. Introspection could also allow queries like “find a sheet by name” or “get all shape IDs on slide 5” which could be added to the interface as needed.

**Leverage Apache POI:** The implementation of these interfaces will heavily rely on Apache POI, which is a robust Java library for Microsoft Office formats. For Excel, POI’s `XSSFWorkbook` (for .xlsx) or `HSSFWorkbook` (for legacy .xls) can be used under the hood. The `ExcelDocumentModel` implementation would wrap a `Workbook` instance. Methods like `setCell` would use POI calls: find the `Sheet` by name, get or create the `Row`, then the `Cell`, set its value (using the appropriate POI method based on the object type, e.g. numeric, string, boolean, or formula). If a formula is set or a cell that depends on a formula is changed, we could call POI’s `FormulaEvaluator` to recalculate and get the new value ([Formula Evaluation - Apache POI](https://poi.apache.org/components/spreadsheet/eval.html#:~:text=The%20POI%20formula%20evaluation%20code,in%2C%20or%20created%20in%20POI)), unless we offload formula calc to the Scala core. For PowerPoint, POI’s `XMLSlideShow` class can open a .pptx and provide slide objects. We can manipulate text via `XSLFTextShape` for text boxes, add images via `XSLFPictureShape`, etc. All these POI operations typically occur in-memory and then the file can be written out when needed.

One challenge to consider is **performance and memory**: POI is known to load entire documents into memory. It can handle quite large files (up to Excel’s inherent limits, e.g. 1,048,576 rows, 16k columns) as long as enough heap is available, but large files consume a lot of memory ([Apache POI™ - HSSF and XSSF Limitations](https://poi.apache.org/components/spreadsheet/limitations.html#:~:text=,very%20large%20amount%20of%20memory)). The design of `ExcelDocumentModel` can mitigate this by using POI’s streaming API (`SXSSFWorkbook`) if we only need sequential write, but for an interactive editor we likely need random access to cells, so the full in-memory model is appropriate. We must be mindful of memory usage; for instance, avoid holding onto too many Change objects or multiple copies of data. The `pullRecentChanges()` method is designed to clear the change log once the server has broadcast them, preventing unbounded growth.

**Extensibility:** By using interfaces, we make the system extensible. For example, if in the future we want to integrate a different library or service for documents (say, using Microsoft’s Graph API to manipulate Excel online, or a cloud service for PowerPoint), we could implement the same interfaces with calls to that service instead of POI. Or we might add a `WordDocumentModel` implementing `DocumentModel` with methods for paragraphs, etc., and the rest of the MCP architecture (protocol, session management) can handle it similarly. The action handlers and protocol can remain generic, relying on the `DocumentModel` abstraction. For now, focusing on POI leverages a proven library – it provides a wide range of features (reading/writing Office files, evaluating Excel formulas, adding slides and formatting shapes, etc.) that we can tap into. By designing our own interfaces on top, we ensure we only expose what the MCP server needs and can optimize or change implementations behind the scenes without affecting the rest of the system.

**Example Usage:** To illustrate, here’s how an action handler might use these interfaces in practice. Suppose an action comes in to change a cell:

```java
// Pseudocode inside an Excel action handler:
public void handleUpdateCell(Session session, String sheetName, int row, int col, String newValue) {
    ExcelDocumentModel model = session.getExcelModel();
    Change change = model.setCell(sheetName, row, col, newValue);
    // change now contains info like type="CellUpdate", cell=A1, newValue, etc.
    // Log or store the change if needed (though model already did internally).
    // Prepare an event to broadcast to clients via protocol (maybe as JSON):
    String eventJson = "{ \"type\": \"CellUpdate\", \"sheet\": \""+sheetName+"\", "
                     + "\"cell\": \""+CellRef.of(row,col)+"\", \"value\": \""+newValue+"\" }";
    serverProtocol.broadcast(session.getId(), eventJson);
}
```

Similarly, for PowerPoint, if an action says “add a new slide”, the handler would call `powerPointModel.addSlide(index, layout)` and get back a `Change` object with the slide number, which can be turned into a JSON event to notify clients that a new slide was added. This demonstrates how the **document model interfaces simplify the action handlers** – they encapsulate the low-level file operations (handled by POI) and present a clean API for making changes and getting results.

## 3. Integration Patterns (Java MCP & Scala XLCR Core)
The existing Scala-based **XLCR core** (presumably responsible for advanced logic like complex formula calculations, business rules, or coordination across documents) needs to interoperate with the new Java MCP server. There are multiple integration patterns to consider, each with trade-offs in complexity, latency, and memory usage. We focus on strategies that ensure **low latency** and **memory efficiency**, so real-time collaboration isn’t bottlenecked by cross-process communication.

**Option 1: In-Process Direct Integration** – Since Scala runs on the JVM and is largely interoperable with Java, one straightforward approach is to incorporate the Scala XLCR core as a library within the Java MCP server process. The Java server can invoke Scala classes and methods directly as if they were Java classes (Scala compiles to JVM bytecode, and standard interoperability allows calling Scala code from Java and vice-versa) ([

This yields very low latency (just in-memory function calls) and avoids any serialization costs. For example, if the XLCR core has a Scala function `recalculateWorkbook(Workbook wb)`, the Java `ExcelDocumentHandler` could directly call that after each cell update, passing the POI `Workbook` object. The Scala code could operate on the POI object or a converted representation. Because Scala collections and Java collections need conversion, some care is needed (e.g. Scala’s `Seq` vs Java’s `List`), but these conversions are trivial compared to network overhead ([

If the Scala core uses advanced features (futures, Akka actors, etc.), it can still be harnessed internally. We may create Java-facing adapter classes or interfaces in Scala if needed to make the API more Java-friendly (for instance, if the Scala code expects Scala `Option` or `Try` types, we can convert those to Java `Optional` or exceptions) ([

The key point is this approach treats the Scala component like a module in the same application. Memory is shared, so no duplication of document data is required – the Scala core could even operate directly on the Java document model’s data structures if they are compatible or if we provide an access API. This is optimal for performance (no context switching or IPC) and should be used if the coupling is acceptable and if the Scala code can run reliably in the same process.

One must ensure that any differences in language semantics (e.g. Scala’s use of implicit parameters or different threading models) are handled. Also, if the Scala core is large or has heavy resource usage, embedding it might increase the memory footprint of the Java server significantly. If the combined system becomes too large or complex to manage as a single process, then we consider out-of-process integration.

**Option 2: Inter-Process Communication (IPC) via RPC** – In this pattern, the Java MCP server and the Scala XLCR core run as separate processes or services, and communicate over an IPC mechanism such as RESTful HTTP calls, gRPC, or a message queue. To maintain low latency and efficiency, a binary RPC protocol like **gRPC** is highly recommended over plain REST. gRPC (which supports JVM on both ends) would let us define service interfaces in a proto file and generate both Java and Scala stubs. It uses HTTP/2 under the hood and Protocol Buffers for payloads, yielding very fast, compact communication ([gRPC vs REST speed comparison](https://blog.shiftasia.com/grpc-vs-rest-speed-comparation/#:~:text=,based%20formats)). In fact, benchmarks show gRPC can outperform REST by a large margin in throughput and latency due to these optimizations ([gRPC vs REST speed comparison](https://blog.shiftasia.com/grpc-vs-rest-speed-comparation/#:~:text=Time%20taken%20,30%20799%20ImageImage)) ([gRPC vs REST speed comparison](https://blog.shiftasia.com/grpc-vs-rest-speed-comparation/#:~:text=,based%20formats)). For our use case, we could define an RPC service, e.g. `XlcrService`, with methods such as `ComputeFormulas(request)` for Excel or `GenerateSlidePreview(request)` for PowerPoint. The Java server, upon certain triggers, would serialize needed data into the request (using protobuf) and call the RPC, getting a response back with results.

To keep memory usage efficient, the design of the RPC calls should be coarse-grained if possible. For example, instead of sending the entire workbook on every edit (which would be very expensive), we might send just the minimal information needed. If the Scala core is responsible for Excel formula recalculation, one strategy is to send a list of (cell coordinates, new values) that changed, and let Scala compute the affected formulas and return just those results. Another strategy is to have the Scala core maintain its own copy of the document state (which it updates incrementally), but that risks duplication of state across processes and increased memory use. A compromise is to initialize the Scala service with the document once (send over the full workbook at session start or have it load from a shared source), then send only deltas on each edit. The data serialization should use protobuf’s efficient binary format to minimize payload size, especially for large documents or many changes.

Alternative IPC methods include using a message broker or event bus (like Kafka, RabbitMQ) where the Java server publishes events and Scala subscribes, or vice versa. However, those typically introduce more latency and complexity, so gRPC (synchronous RPC) or an asynchronous but direct socket communication would be simpler and faster. Another lightweight approach is using a local domain socket or shared memory queue (there are libraries for IPC sockets on JVM ([sbt/ipcsocket: IPC: Unix Domain Socket and Windows ... - GitHub](https://github.com/sbt/ipcsocket#:~:text=sbt%2Fipcsocket%3A%20IPC%3A%20Unix%20Domain%20Socket,Socket%20as%20the%20API))), which avoids TCP overhead if both processes are on the same machine. This could be over-optimization unless we truly need to shave off every millisecond – gRPC over loopback is usually fine given HTTP/2 efficiency.

**Option 3: HTTP/REST Integration** – This is basically a subset of Option 2 where the Java server would call a Scala HTTP REST API (or vice versa). This is easier to implement (since even a simple JSON-over-HTTP can be done with standard libraries, and perhaps the Scala core already exposes some REST endpoints). But it’s less efficient: JSON serialization is text-based and bulkier than protobuf, and HTTP/1.1 request/response for each call adds overhead. Still, for certain interactions that are infrequent or not latency-critical (e.g. maybe saving a file or fetching some analytical data after edits), REST could be acceptable. If going this route, use JSON carefully (perhaps a concise structure for data) and consider HTTP/2 to at least reduce connection overhead. Ensure that calls are designed to avoid chatty behavior – e.g. don’t require 10 REST calls for a single user action; batch data if needed.

**Data Sharing Considerations:** If the Scala core needs access to the entire document (for example, an advanced consistency check on the Excel workbook), an alternative integration is to share the file via the filesystem or a database. For instance, the Java server could save the current workbook to a temp file and instruct the Scala process to load it. This is quite slow (disk I/O) and not truly real-time friendly, but could be used for infrequent tasks (like a heavy batch operation triggered on demand). A variant would be memory-mapped files or shared memory segments to exchange data – this is complex on the JVM and not commonly done for structured data like Office documents. Therefore, passing data through function calls or network messages is more straightforward.

**Recommended Approach:** Given the requirement for **low-latency**, the best approach is likely a hybrid of Option 1 and Option 2. If it’s feasible to run the Scala XLCR logic in-process, that should be done for immediate responsiveness. Scala-Java interoperability is generally very good – “using Scala code in Java is mostly seamless” ([java - Does Kotlin provide any performance boosts? - Stack Overflow](https://stackoverflow.com/questions/44081105/does-kotlin-provide-any-performance-boosts#:~:text=45)) – especially if the Scala code is designed with Java interop in mind (avoiding Scala-specific constructs that don’t translate). We can include the Scala core as a dependency (JAR) in the Java project. The Java server can initialize any needed Scala objects at startup (for example, an `XlcrEngine` singleton) and call methods on it directly for each relevant event. This would yield near-zero serialization cost; data structures can even be shared by reference. We must document the thread-safety: if the Scala core isn’t thread-safe, the integration may need synchronization (e.g. only call it from one thread at a time or use an actor model). But since our server likely already sequences actions per session, we could also funnel requests to Scala accordingly.

If for organizational or stability reasons the team prefers to keep the Scala core as a separate service (perhaps it’s reused by other systems or runs on a different host), then gRPC is the next best choice. We would define proto messages for the types of data we need to send (e.g. cells, slides, etc.) and ensure efficient encoding. Both the Java and Scala sides can use the proto-generated code to handle serialization. The network overhead on localhost with HTTP/2 is minimal – plus gRPC can keep a persistent connection alive, so calls are very fast. The Scala service could be an Akka HTTP or gRPC server that listens for requests from our Java server. We should also implement some backpressure or concurrency control: e.g. if the Java side sends many rapid requests, ensure we don’t flood the Scala side. Possibly aggregate multiple small edits into one RPC call if they happen in a burst.

**Integration use cases:** To make this concrete, consider some specific interactions:

- *Excel formula calculation:* Suppose after every edit to a cell, we want to recalc dependent formulas. We have two choices: use POI’s built-in formula evaluator in Java, or use a Scala logic (maybe XLCR core has a more advanced or optimized calculator). If using Scala, the Java model could send the cell change to Scala and get back a set of (cell, newValue) pairs for all affected formula cells. This could be done synchronously via direct call or RPC. The result is then applied to the Java `ExcelDocumentModel` (updating those cells). The latency must be low to not delay the user interface. If the dataset is large, we might batch multiple edits then recalc once (trade-off between immediacy and throughput).

- *PowerPoint content generation:* Perhaps the Scala core can generate slides automatically (say, based on some data or template). The Java server could offload a request “generate 3 slides about sales data” to Scala. Scala returns a description or even actual PPT content (maybe as XML or by directly editing the file). The Java model then merges that. These kinds of operations might be less frequent and could tolerate a bit more latency, but still we aim for under a second perhaps.

- *Persistence or cross-session analytics:* The Scala pipeline might also handle saving documents or indexing them. In such a case, the Java server might send the final document to Scala for archival when a session closes. Since that’s offline from the user’s perspective, it could be done via a background job or message queue.

**Memory Efficiency:** A critical point when using RPC/IPC is to avoid unnecessary copying of large documents. If the Scala side needs the entire file, sending it repeatedly is wasteful. It would be better to send an identifier or reference. One pattern is to use a **shared data store**: the Java server could put the file (or partial data) in a fast store (like an in-memory grid or Redis or simply on a shared disk location), and then send a message to Scala with the reference (file path or key). The Scala side then reads from that shared source. This decouples the transfer from the control message. However, this adds complexity and might not be needed if we can partition responsibilities such that each side only needs certain info. We should profile what parts of the workload need Scala. If it’s primarily formula math or slide design logic, we can send only those inputs (numbers, text, etc.) and get results.

**Concurrency and Fault Tolerance:** When integrating two systems, consider what happens if the Scala core is slow or crashes. The Java MCP server should be resilient – e.g. if an RPC call times out, perhaps fall back to a simpler built-in mechanism or at least log an error and continue serving basic edits. The design could include asynchronous calls: the Java server might not block the user action waiting for Scala if that’s not critical. For example, for formula updates, maybe apply the user’s edit immediately and later (even a few hundred ms) update the formula results when Scala responds. This way the UI isn’t frozen. Using futures or callbacks for RPC results can help. If running in-process, then exceptions thrown by Scala code need to be caught so they don’t crash the whole server – wrap calls in try/catch and handle gracefully.

In summary, the integration should aim to feel seamless: **Java and Scala components working together as one system.** Thanks to the common JVM platform, direct integration is possible and ideal in terms of performance ([java - Does Kotlin provide any performance boosts? - Stack Overflow](https://stackoverflow.com/questions/44081105/does-kotlin-provide-any-performance-boosts#:~:text=45)). If using a service boundary, then lean on a high-performance RPC like gRPC which “provides better performance and reduced latency by using HTTP/2 and binary Protocol Buffers, compared to text-based REST” ([gRPC vs REST speed comparison](https://blog.shiftasia.com/grpc-vs-rest-speed-comparation/#:~:text=,based%20formats)). Keep data transfer lean (send only what’s needed in compact form) and maintain a robust protocol for requests/responses between the Java MCP and Scala XLCR core.

## 4. Performance Benchmarking Strategy
To ensure the Java-based MCP implementation meets or exceeds the performance of the previous Kotlin version, a comprehensive benchmarking strategy is needed. We will establish **key performance indicators (KPIs)**, create test scenarios to measure them, and use appropriate tools to compare results between the Kotlin and Java implementations. The critical KPIs include: **document load time**, **memory usage**, **edit response latency**, and **throughput under concurrent sessions**.

Each of these KPIs addresses a different aspect of performance:
- **Document Load Time:** The time it takes to open an Excel or PowerPoint file on the server (from request to ready state). This affects the user’s wait time when starting a session.
- **Memory Usage:** The heap footprint of loading and editing a document, which impacts scalability (how many sessions can run in parallel without GC issues or OOM).
- **Response Latency (Edit/Save):** The round-trip time for an edit action – from when a user action is sent to the server to when the client receives confirmation or the update. Also, the time to save a document to disk at session end. Low latency is crucial for real-time feel.
- **Throughput and Concurrency:** The number of edits per second the server can handle and how performance scales as multiple sessions or multiple concurrent users are active. Throughput is important for server-side scalability (e.g. if 50 users each make rapid updates, can the server keep up).

**Benchmark Scenarios:** We will design specific test scenarios to measure each KPI:
- *Load Time:* Use a set of test documents of varying sizes (e.g. a small 100KB Excel, a medium 5MB Excel with many sheets, a large 50MB Excel, similarly small/large PPT files with many slides). Programmatically measure the time from calling `ExcelDocumentModel.open(file)` to the time it returns ready, or from the client perspective, the time from “open” request to the first document data broadcast. We will record this for both Java (new implementation) and Kotlin (old implementation) to compare. If the Kotlin version had known timings, use those as baseline; if not, we can run the same test on the Kotlin system. The expectation is that Java with POI is likely similar or possibly faster if optimized, but we need to verify. Keep in mind POI’s overhead – large Excel files might take a couple of seconds to load if very large due to XML parsing, so results will inform if any optimizations (like streaming) are needed for extremely large files.

- *Memory Usage:* Monitor the Java process’s memory after opening documents and during editing. We can use profiling tools (e.g. VisualVM, JConsole, or attach a JVMTI agent) or simply use `Runtime.getRuntime().totalMemory()` and `freeMemory()` deltas in code to estimate usage. Key measurements: memory used right after loading a doc, memory after a series of edits (to see if there are leaks or large transient allocations), memory after closing a doc (to ensure it’s freed). Apache POI’s documentation notes that large files require large memory and suggests using streaming for very huge files ([Apache POI™ - HSSF and XSSF Limitations](https://poi.apache.org/components/spreadsheet/limitations.html#:~:text=,very%20large%20amount%20of%20memory)). For our use-case, we should note how much RAM a typical file uses. For example, if a 5MB .xlsx uses 50MB RAM, that’s a 10x overhead; if that’s acceptable given our server specs, fine, but if not we might adjust. We should also compare with the Kotlin implementation’s memory (which presumably also used some Office library or similar logic). If the Kotlin version used a different approach (maybe it used POI as well through Kotlin, since Kotlin on JVM could use POI), the memory usage might be nearly identical. Indeed, since both Java and Kotlin compile to bytecode, one wouldn’t expect huge differences in runtime performance or memory for the same logic – “Kotlin’s performance is in most cases the same as Java’s, as it generates similar bytecode” ([java - Does Kotlin provide any performance boosts? - Stack Overflow](https://stackoverflow.com/questions/44081105/does-kotlin-provide-any-performance-boosts#:~:text=45)). Therefore, baseline memory and speed should be comparable; any differences will likely come from library usage patterns or GC behavior that we can fine-tune.

- *Edit Response Latency:* To measure latency of an edit, we can instrument the server to timestamp when a message is received and when the update is sent out. For example, the action handler can record `t1 = System.nanoTime()` on receive and `t2 = System.nanoTime()` after broadcasting to clients, and log the delta. We should gather statistics (min, max, average) for various scenarios: single user editing (no contention), two users in same session editing alternately, many small edits vs a few large operations. We also consider the network overhead: if testing on a local network or a single machine, latencies will be low (a few ms). We might want to simulate a more realistic environment (maybe adding 50ms network delay to see how it holds up, though that’s external). The focus is on server processing latency – ideally this is well under 100ms for normal actions, so that even with network the total is maybe ~100-200ms, which feels real-time. If we find any outliers (like a particular action taking 500ms on server because, say, recalculating a complex formula chain or rendering a large slide), we identify those and consider optimization or asynchronous handling.

- *Throughput & Concurrency:* We need to test how the server performs with multiple simultaneous activities. For example, 10 concurrent sessions (documents) each with one active user, vs 1 session with 10 active collaborators. Both are relevant. We can use load testing tools or custom scripts. One approach is to write a small **load generator** that uses the WebSocket or SSE API to simulate users. For instance, using JMeter with a WebSocket plugin or a Gatling test script to open N WebSocket connections and send edits on each. There are JMeter WebSocket samplers available that can help automate sending messages and measuring response times ([How to test performance of your WebSocket connections using JMeter](https://blog.emumba.com/how-to-test-performance-of-your-websocket-connections-using-jmeter-673c0f649083#:~:text=Create%20a%20Test%20Plan%3A%20,of%20users%20and%20their%20behavior)). We could also write a Java client that spawns threads each simulating a user (connecting to the server and performing a sequence of edits). Key metrics to capture: the server’s CPU usage at high load, any message backlog or queueing, and if throughput (edits/sec processed) scales linearly with number of users up to a point. We expect the Java implementation to handle at least as many concurrent sessions as Kotlin. If the Kotlin version was thread-per-connection or using coroutines, and Java version might use say an NIO framework or threading, their concurrency model might differ. We should ensure the Java server uses asynchronous I/O (e.g. using a WebSocket library like Netty or Spring WebFlux) or a thread pool sized correctly to handle many connections without thread explosion. Benchmark both single-document multi-user throughput and multi-document scenario.

**Baseline Comparison:** It’s important to gather baseline metrics from the Kotlin implementation for reference. For example, if the Kotlin MCP could open a 1MB Excel in 500ms and handle 20 edits/sec per session with 5 sessions, we want to meet or beat that. If direct measurements of the Kotlin system are not readily available in documentation, we can set up the old system in a test environment and use the same tools (JMeter, etc.) to measure it. This gives a ground truth to compare the Java version against. Given that Java and Kotlin on the JVM have similar performance ([java - Does Kotlin provide any performance boosts? - Stack Overflow](https://stackoverflow.com/questions/44081105/does-kotlin-provide-any-performance-boosts#:~:text=45)), we don’t expect major differences solely due to language, but differences could arise from how code is structured (for instance, the Kotlin version might use a different library or have less optimal code in places – or vice versa, it might be highly optimized). So we use the baseline to identify any regressions. If the Java version is initially slower in some aspect, we can profile to find bottlenecks (maybe excessive object creation in POI, or lock contention in session management) and then optimize (e.g. reuse objects, tune GC, or adjust algorithms).

**Benchmark Tools and Setup:** We will use a combination of unit-level benchmarks and system-level benchmarks:

- For micro-benchmarks (like measuring the cost of applying 1000 cell updates or the speed of JSON serialization), we can use **JMH (Java Microbenchmark Harness)**. JMH allows writing small benchmarks that run in a loop with proper JVM warm-up and statistically sound measurements. For example, write a JMH test that loads a fixed small Excel file and measures the `toJson()` method performance or the `setCell` method performance. This can help optimize those hot methods (and compare with Kotlin by perhaps writing an equivalent in Kotlin or using the Kotlin implementation’s code if accessible).

- For end-to-end testing, use **Apache JMeter** or **Gatling**. JMeter with the WebSocket Sampler plugin can simulate many WebSocket clients sending messages and measure response times. Gatling (written in Scala, ironically) has good support for WebSocket scenario testing as well. We could also create a custom test harness in Java that uses multiple threads or async HTTP clients to connect via SSE and measure events (for SSE, we measure how quickly events arrive after an action is injected). Another helpful tool could be a simple latency logger on the client side – e.g. modify a test client to timestamp before sending and after receiving an update.

- For memory, using a **profiler** during tests will be insightful. We can use VisualVM or Java Flight Recorder while running a heavy scenario to see memory usage and CPU hotspots. This isn’t exactly a benchmark but helps ensure our code doesn’t do anything pathologically slow under load.

- We should also consider a **baseline environment**: run tests on a controlled machine (same hardware, same JVM settings for both Kotlin and Java versions) to have apples-to-apples results. Use the same test documents and scenarios on both. Document the JVM parameters (heap size, etc.) used during tests. It might be useful to test with different heap sizes to see if GC pauses affect latency (e.g. if POI causes a lot of garbage, a smaller heap might trigger frequent GC, affecting latency spikes).

**KPI Targets:** Based on typical real-time collaboration requirements, we can set some target thresholds (these can be adjusted after baseline measure): e.g., **Load Time** – an average spreadsheet (~1MB, few thousand cells) should load in <1 second; a large one (~10MB) in <5 seconds. **Latency** – simple edits should have server processing time <50ms on average (not counting network). **Throughput** – the server should handle e.g. 50 edits per second in one session without queue buildup, and scale to, say, 20 concurrent sessions with at least 5 edits/sec each (total 100 ed/s) without significant degradation. These numbers will vary with hardware, but having concrete goals helps validate the design. If current Kotlin version already meets certain numbers, we aim to match or improve them. If not known, we use these as reasonable targets and then refine.

**Benchmark Results and Comparison:** Finally, we will compile the results of our benchmarks in a comparative report. For each KPI, present the Java vs Kotlin measurements (perhaps in a table or graph form). For example, a table of “Document Load Time: Kotlin vs Java” for various file sizes; a chart of “Latency distribution for 1000 edits” for each. This will highlight any regressions to fix and verify the success of the transition. We expect, for instance, that the memory usage of Java+POI might be somewhat higher than an optimized Kotlin solution if that used a lighter library – but if both used POI, they should be similar. We also anticipate that the Java version can be tuned more easily for throughput if we use efficient concurrency libraries. The outcome of benchmarking will guide final optimizations (like if we find SSE connections are slower than WebSocket for our use, we might adjust how we use them, etc.).

In summary, the performance benchmarking strategy is to **define KPIs, simulate realistic usage, measure with proper tools, and iteratively optimize**. By doing so, we ensure the new Java MCP stack is robust and performant. Crucially, because the languages compile to similar JVM code, algorithmic and architectural efficiency will determine performance more than language choice ([java - Does Kotlin provide any performance boosts? - Stack Overflow](https://stackoverflow.com/questions/44081105/does-kotlin-provide-any-performance-boosts#:~:text=45)). With a sound architecture (as described) and careful tuning verified by benchmarks, the Java implementation will confidently support real-time Excel and PowerPoint editing at scale.
